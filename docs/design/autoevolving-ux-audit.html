Auto-Evolving 和 Optimization Events 现存的
9 处 UX 摩擦。

这两个页面承担 SkillForge 飞轮最重要的"显示进化在发生"的责任。但当前实现把同一组数据反复呈现，把核心资产埋到底部，并要求用户记住 15 个 stage 才能用过滤器。下面是按严重度排序的诊断、对应的位置、和最小可行改法。

01 /主页是 7 块平铺，没有视觉重心

页面从上到下连续 7 个独立 section（Header → KPIStrip → Approvals → SignalPanels → WorkflowRunsPanel → AnomalyPanel → EvolveTrajectoryPanel + HarvestedScenarios），每块都用同样规格的卡片包裹，间距统一为 24px。用户进来不知道"现在最该看哪里"。这违反 Bento 布局的初衷——bento 该靠尺寸/密度/位置形成层次，而不是 7 个等大方格。

Critical Effort 中 · 1.5d

没有 hero 区，没有"现在状态"陈述

AutoEvolving.tsx:97-194 · autoevolving.css .ae-page (gap:24px 全局)
现状（每个 ▢ 同等视觉权重）： ┌──────────────────────────────────────────────────┐ │ Auto-Evolving [Evolve][Trigger]│ │ Self-improvement signals, workflow runs… │ ├──────────────────────────────────────────────────┤ │ ▢ Running │ ▢ Done │ ▢ Memory │ ▢ Research N/A │ ← 4 个等大 KPI ├──────────────────────────────────────────────────┤ │ ▢ Pending approvals (条件渲染，时有时无) │ ├──────────────────────────────────────────────────┤ │ ▢ Production │ ▢ AutoResearch │ ▢ Memory │ ← 3 列等宽 ├──────────────────────────────────────────────────┤ │ ▢ Workflow runs (560px DAG) │ ← 唯一真正大块 ├──────────────────────────────────────────────────┤ │ ▢ Anomalies │ ├──────────────────────────────────────────────────┤ │ ▢ Evolve trajectory + Harvested scenarios │ ← 核心资产，最底 └──────────────────────────────────────────────────┘
Why
用户登录到这个页面 80% 是想回答两件事：(a) 飞轮今天有没有转，(b) 有没有要我点头的。当前布局对这两件事都没有给出"一眼答案"。
Fix
引入 状态摘要 hero：把 KPIStrip 升级成 1 行陈述句 + 3 个支持指标。例：
"过去 7 天进化了 4 个 agent · 2 个待你点头 · 3 条信号有效"
然后再让 Approvals 在有内容时升级为 sticky 顶栏，Evolve trajectory 上移到 DAG 上方。

02 /同一个数字在屏幕上出现 三次

"Pending memory proposals = 5" 这条数据，在 KPI 卡、Signal 三宫格的 Memory 面板、以及 Pending approvals 区（如果有 memory approval gate）会同时显示。看似冗余备份，实际是认知噪音——用户每次扫描都要重新核对"这三处是同一个数还是不同维度"。

Major Effort 小 · 0.5d

KPI / Signal / Approval 三层语义没切分清

KPIStrip.tsx + SignalPanels.tsx + WorkflowApproveCard

Before3× 同源

┌ KPI ────────────┐
│ 5  Memory pending│  ← 计数
└─────────────────┘
┌ Signal: Memory ─┐
│ • title A       │  ← 列表
│ • title B       │
│ • title C       │
└─────────────────┘
┌ Approvals ──────┐
│ [memory] title A│  ← 卡片
└─────────────────┘

After每层职责唯一

┌ KPI: 删掉 memory 卡 ─┐
│ 改放"飞轮速度"指标   │
└─────────────────────┘
┌ Signal: Memory ─────┐
│ Today  ─── 2 new    │
│ Past   ─── 3 stale  │  ← 趋势 + 状态
└─────────────────────┘
┌ Approvals ──────────┐
│ 仅显示 humanApprove │
│ 阻塞 workflow 的    │  ← 唯一保留"需要操作"
└─────────────────────┘
Fix
让每个区有唯一职责：KPI=趋势速度、Signal=信号源健康度、Approvals=阻塞性人工 gate。Memory 计数本身在 Signal 里就够，KPI 那格换成 Skill rev/week 或 Verified gain %。

03 /占位卡 "Ships in V2" 永久吃掉 1/3 横向预算

Major Effort XS · 1h

SignalPanels 中间一列长期"还没做完"

SignalPanels.tsx:90-101 · grid-template-columns: 1.4fr 1fr 1.4fr
现状：grid 永远占 1fr，渲染一段 "Autonomous research signal is not wired yet." ┌────────────┬──────────────┬────────────┐ │ Production │ Auto-research│ Memory │ │ list… │ "Ships in V2"│ list… │ │ │ 说明文字 │ │ └────────────┴──────────────┴────────────┘ ↑ 占用 25% 横向空间，0 信息量 建议：在 V2 落地前，AutoResearch 退化为头部一颗 V2 · soon 徽章 ┌─────────────────────┬────────────────────────┐ │ Production │ Memory │ │ ▰▰▰▰▰▰ │ ▰▰▰▰▰▰ │ │ row · row · row │ row · row · row │ └─────────────────────┴────────────────────────┘ Auto-research 即将上线 → (单行通知条，0.5 行高)
Why
把"未来功能"做成卡片占位是 SaaS 常见反模式——用户每天进来都看到同一段文案，会形成"这个产品没在迭代"的潜意识印象。
Fix
改 grid 为 2 列（1fr 1fr），把 AutoResearch 卡降级为 单行通知条 + Roadmap 链接。

05 /核心资产 EvolveTrajectory 被埋到页面最底

Critical Effort 小 · 0.5d

飞轮转动的证据反而在 fold 之下

AutoEvolving.tsx:176-179
页面顺序： 1. Header ↑ 看不到关键信息 2. KPI ↑ 3. Approvals ↑ 4. Signal Panels ↑ 5. Workflow DAG ★ 占 560px,需要滚到一半才看见 6. Anomaly ↑ 7. EvolveTrajectory ← 在第一屏外，需滚 ~1400px 才到 8. HarvestedScenarios ← 更深 但产品定位说："Auto-Evolving" 这个页本质是给运营看 "我的 agent 这周进化了没"。trajectory 折线图正是回答这个问题 最直接的视图。
Fix
把 EvolveTrajectory 提到 KPI 下面，做成横向通栏（高度 220–260px 即可），代替当前 KPI strip 的角色。原 KPI 数字直接叠在折线上方（dual-axis pattern）。WorkflowRunsPanel 和 Anomalies 下沉。

06 /OptimizationEvents 把 15 个 stage 平铺在 Select 里

Major Effort 小 · 0.5d

过滤器要求用户记住整套 stage machine

OptimizationEvents.tsx:46-62 · STAGE_OPTIONS = [15]
现状：[Stage filter ▾] 展开是一个 15 项的扁平列表 dispatch_initiated proposal_pending proposal_approved proposal_rejected candidate_generating candidate_ready candidate_failed candidate_created ← candidate_ready vs created 用户根本分不清 ab_running ab_passed ab_failed canary_started promoted rolled_back verified 这是后端的 stage machine，原样泄漏到 UI。 建议：分组 + 阶段聚合 ┌ Phase ──────────────────────────────────┐ │ ○ Proposal (initiated / pending / │ │ approved / rejected) │ │ ○ Candidate (generating / ready / │ │ failed / created) │ │ ○ A/B (running / passed / failed)│ │ ○ Rollout (canary / promoted / │ │ rolled_back / verified) │ └─────────────────────────────────────────┘ ── Expert mode: show raw stage ▾ ──
Fix
Stage filter 改为二级：先选 phase（4 个），phase 内再选 stage（默认全选）。把"原生 stage 列表"藏在 Expert mode 切换后面。配合 timeline 表格的 stage 列也用同样分组着色（同 phase 同色相、子 stage 用亮度区分）。

07 /有 15 个 stage 却没有漏斗视图

Major Effort 中 · 1d (新组件)

OptimizationEvents 缺最该有的"转化漏斗"

OptimizationEvents.tsx 整页都是表格 + 过滤
缺失视图：从 proposal → promoted 的漏斗 本周 (2026-W26) proposal_pending ████████████████████ 47 proposal_approved ██████████████ 33 (70%) candidate_ready ████████████ 28 (60%) ab_passed ███████ 17 (36%) promoted ████ 10 (21%) rolled_back █ 2 (4%) ────── net +8 verified gains 附带：哪一段流失最大？(approved → candidate_ready 损失 5 个： 点击展开 → 跳到 candidate_failed 的 5 条 row)
Why
该页核心运营问题是"我们的进化建议有多少最终上线了"。当前需要用户用 stage filter 查 5 次手动算。漏斗是这类页面的标配。
Fix
在表格上方插一行 Funnel 组件，时间窗对齐 timeline 表头，每段宽度 = count、悬停显示流失率、点击下钻到该 stage 的 row。

08 /Approve / Reject 流的断点

Major Effort 小 · 0.5d

Reject 必填理由，Approve 不必填理由——两边不对称

OptimizationEvents.tsx:852-883 (RejectModal)
当前： Approve → 直接通过 (无确认) Reject → Modal 强制输入 reason 问题： • Approve 是更危险的动作(把候选放到 A/B 跑/直接上线), 反而无任何 friction • Reject 的 reason 不会回流到 candidate generator,无法形成学习闭环 • 没有 batch action (5 个 pending 时需要点 10 次) 建议： Approve → inline 确认 (小 popover)，可选写一行 "why approved" → 这条 note 进入下一次候选生成的 hint Reject → 保持必填 + 增加 quick-reasons (Chip 多选： "off-topic" "duplicate" "low-impact" "risk-too-high") Batch → table 顶部勾选行后出现 [Approve all] [Reject all w/ reason]
Fix
对称化两端摩擦 + 让 reject reason 形成可被 evolve agent 读取的反馈语料。这一改动 ROI 极高，是飞轮闭环的关键一环。

09 /AnomalyPanel 只显示症状，不引导动作

Minor Effort XS · 0.5d

异常红条 + 一行 reason，但用户不知道下一步该干嘛

autoevolving.css .ae-anomaly-row · AnomalyPanel.tsx
Fix
每行末尾加 1–2 个 建议动作：Open trace ↗ · Suppress 24h · Trigger investigation workflow。让"红条 → 行动"链路闭合。
Minor Effort XS · 1h

顺手：DAG host 固定 560px 高，小屏上撑不开

.ae-dag-host { height:560px }
Fix
改 min-height:420px; max-height: min(560px, 60vh)，让小屏(1366×768 笔记本)不至于 DAG 把 anomaly 推到 fold 之下。

10 /分轨建议

按"动后端 / 不动后端"切两轨，再按时间分。Quick wins 全部不动 API，1.5 个工作日内可全部上线。
