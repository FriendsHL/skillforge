---

## 1. 核心观点

评测的单元不是单次 tool call，而是**一条完整的 Agent 执行轨迹（trajectory）**。
单步评分意义有限，轨迹级别的 outcome + process 双维度评分才能反映真实质量。

---

## 2. 归因分层

将可能导致失败的原因按层分组，归因时先定层、再定具体原因：

```
【LLM 决策层】
├── Prompt        — system prompt、task 描述、few-shot 示例
├── Tool 定义      — 工具名称、描述、参数 schema（LLM 看到的部分）
└── LLM           — 模型选择、temperature、provider

【执行层】
├── Skill 实现     — tool 被调用后真正运行的代码
├── Hook 脚本      — pre/post tool use、stop hook
└── 框架           — agent loop、重试逻辑、错误处理、turn 管理

【上下文层】
├── 记忆           — 存储内容、检索策略、记忆质量
└── Compaction     — 长上下文压缩策略（信息丢失风险）
```

**关键区分**：
- **Tool 定义 ≠ Skill 实现**：描述写错是 prompt 问题，实现写错是执行问题，归因路径完全不同
- **记忆 ≠ Compaction**：前者是主动存取，后者是被动截断，失败原因不同
- **任务本身**：输入 ambiguous 或目标不可达，此类失败不应归因到任何组件

---

## 3. 归因机制：解耦评判

对每条轨迹里的每个 tool call，分两步独立评判：

```
上下文 → [LLM 决策] → tool_call (name + params) → [Skill 执行] → tool_result
             ↑                                            ↑
          评判点 1                                     评判点 2
```

**评判点 1：LLM 决策质量**

给 Judge 输入 `{上下文, tool_call}`，问：这个调用决策合理吗？

| 信号 | 指向 |
|------|------|
| 选错 tool（该用 A 用了 B） | Tool 描述不清 |
| 参数语义错（类型对但含义偏） | 参数描述 ambiguous |
| 应调未调 | 描述没让 LLM 意识到需要它 |
| 思考过程对工具有困惑 | Tool 描述问题 |

**评判点 2：Skill 执行质量**

给 Judge 输入 `{tool_call, tool_result}`，问：这个输出正确吗？

| 信号 | 指向 |
|------|------|
| 抛异常 / error | Skill 执行问题 |
| 参数合法，结果为空/错 | Skill 逻辑问题 |
| 输出格式与描述声称的不符 | Skill 问题（或描述与实现不一致） |

**诊断矩阵**：

| 决策合理？ | 执行正确？ | 归因 |
|-----------|-----------|------|
| ❌ | 任意 | Tool 描述问题（LLM 决策层） |
| ✅ | ❌ | Skill 执行问题（执行层） |
| ✅ | ✅ | 问题在别处（下一步、记忆、框架） |

## 4. 评测标准分工

| 层级 | 谁来定标准 | 理由 |
|------|-----------|------|
| Outcome（任务成没成） | 用户 | 目标是用户的，只有用户知道"算完成" |
| Process（路径对不对） | 模型为主 + 用户 Rubric | 用户给原则，模型按原则打分 |
| 归因（哪步是 root cause） | 模型为主 | 太细，用户无法手动逐步标注 |

**Rubric 是关键桥梁**：用户不需要标每条轨迹，但需要写清楚评判原则，例如：
- "调用 tool 之前应该先确认 X"
- "Skill 返回空结果时应该重试而不是放弃"
- "超过 N 步没进展算路径失败"

模型拿着 Rubric 打分，用户抽查 5-10% 验证方向对不对。

> **最大风险**：Rubric 本身写错，系统会往错误方向自进化。用户真正需要守护的是 Rubric 的质量，而不是每条样本的标注。

## 5. Ground Truth 自动生成

对于开放性任务（如调研类），ground truth 无法由人工持续维护，采用三方案叠加：

```
外部锚点（GitHub awesome-* / 权威列表）
    + Gold Agent（最强配置多次运行，提取 key entities）
    + 多 Agent 共识（N 个独立 Agent 投票，≥2/3 出现 → 认定为 ground truth）
        ↓
    动态 ground truth（定期更新）
        ↓
    用户抽样校验（发现系统性漂移时介入）
```

**各方案作用**：
- 外部锚点：防止系统性盲区，保底不漂移
- Gold Agent：覆盖最新/长尾内容
- 多 Agent 共识：过滤偶发性噪声

用户不在关键路径，只在"系统跑偏了"时介入。

## 6. 示例：调研型任务的评测设计

以"调研 Agent 相关框架"为例：

**Outcome 评分**：

| 维度 | 评法 |
|------|------|
| 覆盖率 | 对比 ground truth 框架列表，计算召回率 |
| 准确性 | 框架描述有无事实错误（Judge 校验） |
| 结构性 | 有无对比维度、结论、推荐 |
| 可用性 | 用户拿到报告能否做决策 |

**Process 评分**：

| 好轨迹 | 坏轨迹 |
|--------|--------|
| 搜索词有针对性 | 一直重复相似查询 |
| 多源交叉验证 | 只查一次就整理 |
| 发现空白后补查 | 空白直接跳过 |
| 工具选择正确 | 该搜索时去读本地文件 |

**归因路径**：

```
报告覆盖率低
  ├── 搜索词设计差？      → Prompt 问题
  ├── 搜索 Skill 返回少？  → Skill 执行问题
  └── 搜到了但没整合进去？ → LLM 综合能力 / 框架问题

报告有错误信息
  ├── Skill 返回的数据本身错？ → Skill / 数据源问题
  └── 数据对但 LLM 解读错？   → Prompt / 模型问题
```

## 7. 归因 → 改进信号映射

每个归因层对应不同改进动作，自动化程度不同：

| 归因层 | 改进动作 | 自动化程度 |
|--------|---------|-----------|
| Prompt | 重写 system prompt、补充 few-shot 示例 | ✅ 全自动 |
| Tool 定义 | 重写工具描述 / 参数说明 | ✅ 全自动 |
| LLM | 切换模型 / 调整 temperature | 🔶 跑 benchmark 后选 |
| Skill 实现 | 修 bug / 扩展能力 | 🔴 需要 dev agent / 人工 |
| 框架 | 调整重试策略 / turn 上限 | 🔶 配置变更 |
| Hook | 修改触发条件 / 补逻辑 | 🔴 需要代码变更 |
| 记忆 | 调整存储策略 / 检索逻辑 | 🔶 半自动 |
| Compaction | 调整压缩阈值 / 策略 | 🔶 配置变更 |

**改进信号生成流程**：

```
归因结果（哪层，哪类失败，出现频率）
        ↓
聚合：同一层同类失败 → 归为一个改进任务
        ↓
按自动化程度分流：

  ✅ 全自动 → 生成候选改进（如新版 prompt）
              → 用原失败 case 验证
              → 通过 → 自动上线 / 不通过 → 丢回队列

  🔶 半自动 → 生成改进建议 + 配置 diff → 人工确认 → 执行

  🔴 需代码 → 生成 issue / PR draft → 人工 review → 合并
```

**改进验证（必须）**：每次改进上线前跑两个检验：
1. **目标 case 通过**：原来失败的 case 修好了没
2. **回归 case 不退化**：原来通过的 case 还过吗

## 8. 自进化闭环全图

```
轨迹采集
    ↓
Outcome 评分（任务成没成）
    ↓
Process 评分（路径对不对）
    ↓
归因（哪一步是关键 cause）
    ↓
改进信号生成（按归因层分流）
    ↓
验证（目标 case + 回归 case）
    ↓
上线
    ↑                    ↓
    └── 新轨迹 ← 漂移监控 ┘
```
